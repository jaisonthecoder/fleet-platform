# Environment Clone & Database Migration Guide

**Purpose:** move this project — code, schema migrations, and (optionally) the current
database contents — from one machine to another (new PC, new dev's laptop, a fresh
CI/sandbox environment). Written so an AI agent or engineer can follow it without any
other context.

**Audience:** whoever is setting up the repo on a *new* machine, and whoever is
preparing a data export on the *source* machine.

---

## 0. What travels automatically vs what doesn't

| Item | Travels via `git clone`? | Notes |
|---|---|---|
| Source code (`app-api/src`, `app-ui/src`, …) | ✅ Yes | normal tracked files |
| Schema migrations (`app-api/drizzle/migrations/*.sql` + `meta/_journal.json`) | ✅ Yes | **not** gitignored — treat as source code |
| Dev seed script (`app-api/scripts/seed-dev.sql`) | ✅ Yes | idempotent baseline data (roles/lookups/persons) |
| `docker-compose.yml`, `docker/postgres/init/*.sql` | ✅ Yes | recreates empty Postgres+Redis containers identically |
| `.env`, `.env.local`, `.env.uat`, `.env.prod`, `.env.development`, `.env.production`, `.env.ut` | ❌ No — gitignored | must be copied manually (see §2) |
| Actual **database rows** (Postgres data) | ❌ No | lives in a Docker **named volume** on the source machine only (see §4) |
| Redis contents | ❌ No, and **not needed** | pure cache / outbox-poll state, rebuilds itself |
| `node_modules/`, `dist/` | ❌ No, and not needed | regenerated by `pnpm install` / `pnpm build` |

Before starting, confirm the source machine's working tree is clean so nothing is left
behind:

```bash
git status --short   # should print nothing
git push              # if it prints anything, commit/push first
```

## 1. Prerequisites on the new machine

- **Node** `>=24`
- **pnpm** `>=11.15.0` (pinned in every `package.json`'s `packageManager`/`engines`
  field — `pnpm install` / any `pnpm run` script **fails the engine check** below
  this version; install/upgrade pnpm *before* anything else)
- **Docker Desktop** (or another Docker engine) — runs Postgres (TimescaleDB image)
  + Redis
- **git**

```bash
node --version   # expect v24.x or newer
pnpm --version   # expect 11.15.0 or newer
docker --version
```

---

## 2. Clone the repo + restore the env files

```bash
git clone https://github.com/jaisonthecoder/fleet-platform.git
cd fleet-platform
```

Copy these 10 files from the source machine into the same relative paths (secure
channel — several contain Entra/Azure AD client IDs and other config, not just
placeholders):

```
app-api/.env.local
app-api/.env.uat
app-api/.env.prod
app-ui/.env
app-ui/.env.development
app-ui/.env.local
app-ui/.env.production
app-ui/.env.ut
```

(`app-api/.env.example` and `app-ui/.env.example` **are** in git — use them as a
template if you'd rather recreate the files from scratch instead of copying secrets.)

Install dependencies once, from the **repo root** (`pnpm-workspace.yaml` covers both
`app-api` and `app-ui` — do not run separate installs per package):

```bash
pnpm install
```

## 3. Start infrastructure containers

```bash
docker compose up -d
docker compose ps   # wait until fleet-postgres and fleet-redis both show "healthy"
```

This creates two **new, empty** named volumes (`fleet-pgdata`, `fleet-redisdata`) and,
on Postgres's first boot only, runs `docker/postgres/init/01-extensions.sql`
(`CREATE EXTENSION timescaledb, pgcrypto`). The remaining extensions the schema needs
(`ltree`, `btree_gist`) are created later by migration `0000` itself, not by the init
script — so don't skip step 4/5 expecting them to already exist.

Now pick **one** of the two paths below.

---

## 4A. Path A — fresh baseline (recommended default)

Use this when you don't need the exact rows from the old machine — just the schema
and a clean dev-seeded baseline (18 dev persons/roles, ~28 lookups, 1 organization, 0
vehicles).

```bash
cd app-api
export DATABASE_URL="postgres://fleet:fleet@localhost:5442/fleet"
export REDIS_URL="redis://localhost:6379"

pnpm db:migrate   # applies every migration in drizzle/migrations, in order
pnpm db:seed      # runs scripts/seed-dev.sql (idempotent — safe to re-run)
```

Skip straight to §5 (verification).

---

## 4B. Path B — exact data clone (bookings/vehicles/org edits you actually need)

Use this when the source machine has real work-in-progress data (bookings, vehicles,
organization hierarchy edits, policy drafts, etc.) that a fresh seed would lose.

### On the **source** machine

```bash
docker exec fleet-postgres pg_dump -U fleet -d fleet -Fc -f /tmp/fleet.dump
docker cp fleet-postgres:/tmp/fleet.dump ./fleet-backup.dump
```

Transfer `fleet-backup.dump` to the new machine (USB drive, internal file share,
encrypted cloud transfer — it may contain real business data; do not email it or put
it anywhere public).

### On the **new** machine

Postgres must already be up and empty (from §3 above — do **not** run `db:migrate` or
`db:seed` first, the restore brings the fully-migrated schema with it):

```bash
docker cp fleet-backup.dump fleet-postgres:/tmp/fleet.dump
docker exec fleet-postgres pg_restore -U fleet -d fleet --clean --if-exists /tmp/fleet.dump
```

- `--clean --if-exists` drops any conflicting objects before recreating them — safe
  here because the target database is brand new, but **never** run this against a
  database you intend to keep, it will destroy existing data in `fleet`.
- The dump includes the `drizzle.__drizzle_migrations` tracking table, so the
  restored database already reports itself at the source machine's latest migration
  (e.g. `0028`) — no separate migration step is required. Optionally confirm nothing
  is pending:

```bash
cd app-api
export DATABASE_URL="postgres://fleet:fleet@localhost:5442/fleet"
pnpm db:migrate   # idempotent — should report "no changes" / apply nothing
```

## 5. Verify

```bash
# 1. Confirm migration state matches expectations (adjust the "8" filter to taste)
docker exec fleet-postgres psql -U fleet -d fleet -X -q -P pager=off \
  -c "select id, hash, created_at from drizzle.__drizzle_migrations order by id desc limit 8"

# 2. Confirm the schema is real (spot-check a known table)
docker exec fleet-postgres psql -U fleet -d fleet -X -q -P pager=off \
  -c "select to_regclass('fleet.booking') as table_name"
```

Then boot the app:

```bash
cd app-api
pnpm run start     # builds + runs dist/main.js -> API on :3000, Swagger at /api-docs
```

```bash
# separate terminal
cd app-ui
pnpm run dev        # Vite dev server on :5173
```

Checklist:
- API `GET http://localhost:3000/health` → `200`
- API `GET http://localhost:3000/api/v1/dev/users` → returns the dev user list (Path
  A) or your real persons (Path B) — **empty array here almost always means the API
  process crashed on boot**, check its terminal output first, not the frontend
- UI `http://localhost:5173/en/login` → dev-login user list renders, picking a user
  lands in the app shell (not bounced back to `/login`)
- `CORS_ORIGINS` in `app-api/.env.local` includes the UI's actual origin
  (`http://localhost:5173` by default) — mismatches show up as browser CORS errors,
  not 401s

## 6. Known gotchas (carried over from prior setup sessions — check these first)

- **pnpm engine mismatch.** If `pnpm install`/`pnpm run <script>` fails an engine
  check, the machine's `pnpm --version` is below `11.15.0` (or a stray older pnpm
  binary is earlier on `PATH`). Fix the version before touching anything else —
  don't try to work around it by calling tool binaries directly.
- **API silently exits 1 on boot.** Most common cause historically: an optional
  Swagger dependency (`@fastify/static`) declared but not installed. Re-run
  `pnpm install` at the repo root if the API process exits immediately; check its
  stdout/stderr for the actual thrown error before assuming it's a DB/env problem.
- **UI dev server binds `[::1]:5173` (IPv6).** `curl 127.0.0.1:5173` can return
  `000`/connection-refused even though the server is up — use `localhost` or `[::1]`
  when probing it manually.
- **git-bash `node` is winpty-wrapped on Windows.** Running `node` directly in a
  git-bash terminal can print `stdout is not a tty` and swallow logs. Prefer the
  `pnpm run <script>` wrappers (as used throughout this doc) over calling `node`
  directly; if you must call `node` raw, redirect output to a file instead of
  relying on the terminal.
- **`DATABASE_URL` default port is `5432`, this project maps Postgres to `5442`.**
  Always export `DATABASE_URL=postgres://fleet:fleet@localhost:5442/fleet` explicitly
  before running any `db:*` script or integration test — the default silently points
  at the wrong port if you forget.
- **Dev-login only works outside production.** `AUTH_DEV_LOGIN` in `.env.local` must
  be `true` (default) and `NODE_ENV` must not be `uat`/`production`, or `/me` and the
  dev user list will reject requests without a real Entra bearer token.

## 7. Quick reference — all relevant commands in one place

```bash
# Source machine: export data
docker exec fleet-postgres pg_dump -U fleet -d fleet -Fc -f /tmp/fleet.dump
docker cp fleet-postgres:/tmp/fleet.dump ./fleet-backup.dump

# New machine: get the code
git clone https://github.com/jaisonthecoder/fleet-platform.git
cd fleet-platform
# -> copy the 10 .env.* files here manually
pnpm install
docker compose up -d

# New machine, Path A (fresh baseline)
cd app-api
export DATABASE_URL="postgres://fleet:fleet@localhost:5442/fleet"
export REDIS_URL="redis://localhost:6379"
pnpm db:migrate
pnpm db:seed

# New machine, Path B (restore exact data)
docker cp fleet-backup.dump fleet-postgres:/tmp/fleet.dump
docker exec fleet-postgres pg_restore -U fleet -d fleet --clean --if-exists /tmp/fleet.dump

# Run it
cd app-api && pnpm run start   # :3000
cd app-ui  && pnpm run dev     # :5173
```
